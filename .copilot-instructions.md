# Copilot Instructions

## Workspace Structure

This is the **workspace repository** for the Deepstaging multi-repository ecosystem.

**Important**: The parent directory (`~/code/org/deepstaging/`) is **NOT** a git repository. It's just a container where:
- This workspace repo (`workspace/`) lives
- Other repos (`deepstaging/`, `effects/`, etc.) are cloned as siblings
- Users bootstrap the environment by cloning this workspace, then running `scripts/bootstrap.sh`

## File Placement Rules

### Documentation

**Permanent workspace knowledge** → **`.docs/`** (git tracked)
- Convention decisions
- Architecture patterns
- Getting started guides
- Migration guides
- Quick references

**Temporary session notes** → **`.session/`** (gitignored)
- Implementation status
- Planning questions
- Work-in-progress notes
- Session-specific tracking

When in doubt: Start in `.session/`, move to `.docs/` when finalized.

### Agent State
All agent-specific state lives in workspace root:
- `.claude/` - Claude agent memory and context
- `.copilot/` - GitHub Copilot state
- `.cursor/` - Cursor editor state

**Never** create agent directories in sibling repositories (`../deepstaging/`, `../effects/`, etc.).

### Scripts
Cross-repository automation goes in `./scripts/`:
- `bootstrap.sh` - Clone repos, setup environment
- Build/test/publish scripts that operate across multiple repos

## Working with Sibling Repositories

When operating on code in `../deepstaging/` or `../effects/`:
- Those are separate git repositories
- They have their own `docs/` for user-facing documentation (separate from workspace `.docs/`)
- They should not have agent directories
- Their `.gitignore` should block `.claude/`, `.copilot/`, `.cursor/`

## Repository Documentation vs Workspace Documentation

- **Workspace `.docs/`** (here) - Session docs, decisions, cross-cutting knowledge
- **Package `docs/`** (in siblings) - User-facing, versioned documentation that ships with packages
